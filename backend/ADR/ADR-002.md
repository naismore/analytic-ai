# ADR-002: Стратегия аутентификации (JWT + Refresh Token)

**Статус:** Accepted
**Дата:** 2026-03-18

---

## Контекст

Система требует:

* аутентификацию пользователей
* безопасный доступ к защищённым endpoint’ам
* поддержку долгоживущих сессий
* возможность масштабирования (в том числе горизонтального)

Дополнительно:

* backend является stateless
* frontend работает через HTTP API
* используется ASP.NET Core Identity

---

## Проблема

Необходимо выбрать механизм аутентификации, который:

* безопасен
* не требует хранения сессий на сервере
* поддерживает обновление токенов
* хорошо работает в distributed среде

---

## Решение

Используется схема:

### Access Token (JWT) + Refresh Token

* **Access Token**

  * короткоживущий (например, 15–30 минут)
  * используется для доступа к API
  * передаётся в `Authorization: Bearer`

* **Refresh Token**

  * долгоживущий
  * хранится в HttpOnly cookie
  * используется для получения нового Access Token

---

## Архитектурная реализация

* генерация токенов: `ITokenService`
* логин: `AuthService`
* обновление токена: `RefreshTokenCommandHandler`
* хранение refresh токенов: `IRefreshTokenRepository`

---

## Flow

### Login

1. Пользователь логинится
2. backend:

   * проверяет через Identity
   * генерирует access + refresh токены
3. refresh token кладётся в cookie
4. access token возвращается в response

---

### Доступ к API

Клиент отправляет:

```
Authorization: Bearer <access_token>
```

---

### Refresh

1. backend читает refresh token из cookie
2. валидирует его
3. выдаёт новую пару токенов

---

## Последствия

### Плюсы

* stateless backend (нет server-side сессий)
* хорошая масштабируемость
* безопасное хранение refresh token (HttpOnly)
* гибкий контроль времени жизни токенов

---

### Минусы

* усложнение логики (refresh flow)
* необходимость хранения refresh токенов
* нужно аккуратно управлять ротацией

---

### Риски

* компрометация refresh token
* отсутствие ротации токенов
* неправильная настройка cookie

---

## Меры безопасности

* HttpOnly cookie (нельзя прочитать из JS)
* Secure flag (только HTTPS)
* SameSite=Strict
* ограничение срока жизни токенов
* возможность отзыва refresh токена

---

## Альтернативы

### Session-based auth (cookie session)

Почему не выбрана:

* требует хранения состояния на сервере
* плохо масштабируется

---

### Только JWT (без refresh token)

Почему не выбрана:

* плохой UX (частые логины)
* невозможно продлить сессию безопасно

---

### OAuth2 / OpenID Connect

Почему не выбрана:

* избыточно для текущей системы
* усложняет инфраструктуру

---

### Хранение refresh token в localStorage

Почему не выбрана:

* уязвимость к XSS

---

## Итог

Выбрана схема:

* Access Token (JWT)
* Refresh Token (HttpOnly cookie)
* Identity через ASP.NET Core Identity

Решение обеспечивает:

* безопасность
* масштабируемость
* удобство для пользователя

---

## Возможные улучшения

* ротация refresh токенов (rotate on use)
* blacklist / revoke механизм
* device-based sessions
* аудит логинов
