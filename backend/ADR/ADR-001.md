# ADR-001: Выбор архитектурного стиля и технологий

**Статус:** Accepted
**Дата:** 2026-03-18

---

## Контекст

Разрабатывается backend-сервис с функциональностью:

* аутентификация и авторизация пользователей
* управление сессиями рекомендаций
* интеграция с внешним LLM-сервисом
* хранение бизнес-данных и identity-данных

Требования к системе:

* расширяемость (добавление новых фич и интеграций)
* тестируемость бизнес-логики
* изоляция инфраструктуры
* возможность масштабирования
* поддерживаемость кода

---

## Проблема

Необходимо выбрать:

1. Архитектурный стиль
2. Способ организации бизнес-логики
3. ORM и доступ к данным
4. Решение для аутентификации и авторизации

---

## Решение

### 1. Архитектура: Clean Architecture + CQRS

Выбрана архитектура с разделением на слои:

* **WebApi (Presentation)**
* **Application (CQRS + бизнес-логика)**
* **Domain (сущности и интерфейсы)**
* **Infrastructure (реализации, БД, внешние сервисы)**

Дополнительно используется паттерн **CQRS**:

* Commands — изменяют состояние
* Queries — только читают данные

Для маршрутизации используется Mediator (MediatR).

---

### 2. ORM: Entity Framework Core

Используется EF Core для работы с базой данных.

**Причины выбора:**

* нативная интеграция с .NET
* поддержка LINQ
* миграции из коробки
* удобная работа с DbContext
* активная поддержка

**Роль в архитектуре:**

* используется только в Infrastructure
* доступ через Repository pattern
* Application не зависит от EF Core напрямую

---

### 3. Аутентификация: ASP.NET Core Identity

Используется Identity для управления пользователями.

**Особенности:**

* отдельный контекст: `AppIdentityDbContext`
* хранение пользователей, ролей и паролей
* интеграция с токенами через `ITokenService`

**Причины выбора:**

* готовая и безопасная реализация
* поддержка ролей и claims
* минимизация кастомной логики безопасности

---

### 4. Разделение БД

Используются два контекста:

* `DTADbContext` — бизнес-данные
* `AppIdentityDbContext` — Identity

**Причины:**

* изоляция безопасности
* независимые миграции
* возможность масштабирования
* подготовка к возможному выносу Identity

---

### 5. Интеграция с LLM

LLM вынесен в отдельный сервис:

* взаимодействие по HTTP
* реализовано через `ILLMService`

**Причины:**

* слабая связность
* независимое масштабирование
* возможность замены модели

---

## Последствия

### Плюсы

* высокая тестируемость (Application слой изолирован)
* слабая связность между слоями
* гибкость в замене инфраструктуры
* масштабируемость
* понятная структура проекта

---

### Минусы

* повышенная сложность (много слоёв и абстракций)
* больше кода (handlers, интерфейсы, DTO)
* возможный оверхед для простых сценариев

---

### Риски

* избыточное использование CQRS для простых операций
* “анемичная” доменная модель
* рост количества классов

---

## Альтернативы

### Layered Architecture (без CQRS)

Почему не выбрана:

* хуже масштабируется
* сложнее тестировать бизнес-логику

---

### Minimal API + Service Layer

Почему не выбрана:

* плохо подходит для сложной бизнес-логики
* слабая структурированность

---

### Dapper вместо EF Core

Почему не выбран:

* больше ручной работы
* нет миграций
* хуже интеграция с доменной моделью

---

### Самописная аутентификация

Почему не выбрана:

* высокий риск ошибок безопасности
* дублирование готовых решений

---

## Итог

Выбран стек:

* Архитектура: Clean Architecture + CQRS
* ORM: Entity Framework Core
* Auth: ASP.NET Core Identity
* Integration: HTTP (LLM service)

Решение ориентировано на:

* масштабируемость
* поддерживаемость
* безопасность
