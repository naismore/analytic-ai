##  ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### Модуль 1: Управление пользователями

**F01:** Пользователи должны иметь возможность начать подбор без регистрации.

**F02:** Пользователи должны иметь возможность зарегистрироваться по email для сохранения истории подборов.

**F03:** Пользователи должны иметь возможность авторизоваться через email/пароль.

**F04:** Пользователи должны иметь возможность восстановить пароль через email.

**F05:** Система должна сохранять историю подборов для зарегистрированных пользователей.

**F06:** Пользователи должны иметь возможность просматривать историю своих подборов.


### Модуль 2: Опрос и сбор требований

**F07:** Система должна проводить опрос пользователя из 5-7 вопросов.

**F08:** Система должна предоставлять разные типы вопросов (один выбор, множественный выбор, шкала).

**F09:** Система должна отображать прогресс прохождения опроса.

**F10:** Пользователи должны иметь возможность вернуться к предыдущему вопросу.

**F11:** Пользователи должны иметь возможность пропустить вопрос.

**F12:** Система должна предоставлять подсказки и примеры к сложным вопросам.

**F13:** Система должна запоминать прогресс опроса при перерыве (для зарегистрированных).

**F14:** Система должна адаптировать вопросы в зависимости от предыдущих ответов.

### Модуль 3: Критерии подбора

**F15:** Система должна учитывать уровень опыта пользователя (новичок/средний/продвинутый).

**F16:** Система должна учитывать объем обрабатываемых данных (<1ГБ/1-100ГБ/>100ГБ).

**F17:** Система должна учитывать типы задач (визуализация, ML, ETL, статистика).

**F18:** Система должна учитывать бюджетные ограничения (бесплатно/до $100/не ограничен).

**F19:** Система должна учитывать технический бэкграунд (гуманитарий/технарь/разработчик).

**F20:** Система должна учитывать потребность в командной работе.

**F21:** Система должна учитывать требования к интеграции с другими системами.

**F22:** Система должна учитывать требования к безопасности.

### Модуль 4: Генерация рекомендаций

**F23:** Система должна генерировать минимум 3 рекомендации инструментов.

**F24:** Система должна четко выделять основной рекомендуемый инструмент.

**F25:** Система должна предоставлять альтернативные варианты (2-3 инструмента).

**F26:** Система должна показывать процент соответствия для каждого инструмента.

**F27:** Система должна предоставлять обоснование для каждой рекомендации.

**F28:** Система должна указывать сильные стороны инструмента для данного кейса.

**F29:** Система должна предупреждать о возможных сложностях и ограничениях.

**F30:** Система должна предлагать комбинации инструментов (стек технологий).

### Модуль 5: Детальная информация об инструментах

**F31:** Система должна предоставлять детальную информацию о каждом инструменте.

**F32:** Информация должна включать описание, категорию, цену, платформы.

**F33:** Информация должна включать требования к обучению и сложность освоения.

**F34:** Информация должна включать примеры использования и кейсы.

**F35:** Информация должна включать обучающие материалы.

**F36:** Информация должна включать отзывы других пользователей.

### Модуль 6: Сравнение инструментов

**F37:** Система должна позволять сравнивать 2-3 инструмента.

**F38:** Сравнение должно отображаться в виде наглядной таблицы.

**F39:** Сравнение должно включать ключевые параметры (цена, сложность, функции).

**F40:** Система должна выделять преимущества каждого инструмента для данного кейса.

**F41:** Пользователи должны иметь возможность выбирать параметры для сравнения.

### Модуль 7: Дорожная карта обучения

**F42:** Система должна генерировать дорожную карту обучения для рекомендуемых инструментов.

**F43:** Дорожная карта должна включать пошаговый план освоения.

**F44:** Дорожная карта должна включать оценку времени на каждый этап.

**F45:** Дорожная карта должна адаптироваться под уровень пользователя.

### Модуль 8: Экспорт и сохранение результатов

**F46:** Система должна генерировать уникальную ссылку для доступа к результатам.

**F47:** Система должна позволять экспортировать результаты в PDF.

**F48:** Экспортированный PDF должен включать все рекомендации с обоснованиями.

**F49:** Экспортированный PDF должен включать сравнительную таблицу.

**F50:** Экспортированный PDF должен включать дорожную карту обучения.

### Модуль 9: Обратная связь и улучшение

**F51:** Пользователи должны иметь возможность оценить полезность рекомендаций.

**F52:** Пользователи должны иметь возможность оставить комментарий к рекомендациям.

**F53:** Система должна собирать анонимную статистику использования.

### Модуль 10: Администрирование

**F62:** Администраторы должны иметь возможность добавлять новые инструменты.

**F63:** Администраторы должны иметь возможность редактировать информацию об инструментах.

**F64:** Администраторы должны иметь возможность удалять инструменты.

**F65:** Администраторы должны иметь возможность редактировать правила подбора.

**F66:** Администраторы должны иметь возможность просматривать статистику использования.

**F67:** Администраторы должны иметь возможность управлять пользователями.

## НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### Производительность 

**PER01:** Время загрузки главной страницы не должно превышать 3 секунды.

**PER02:** Время отклика API при генерации рекомендаций не должно превышать 5 секунд.

**PER03:** Время перехода между вопросами опроса не должно превышать 1 секунду.

**PER04:** Время генерации PDF-отчета не должно превышать 5 секунд.

**PER05:** Время поиска по базе инструментов не должно превышать 1 секунду.

**PER06:** Время отображения результатов после завершения опроса не должно превышать 3 секунды.

**PER07:** Система должна поддерживать не менее 100 одновременных пользователей на начальном этапе.

**PER08:** Максимальное время простоя без потери сессии должно составлять 30 минут.

### Масштабируемость 

**SCA01:** На момент запуска ожидается не менее 500 активных пользователей в месяц.

**SCA02:** Через год ожидается рост до 5 000 активных пользователей в месяц.

**SCA03:** Через 3 года ожидается рост до 20 000 активных пользователей в месяц.

**SCA04:** Система должна выдерживать пиковую нагрузку до 500 одновременных запросов.

**SCA05:** Архитектура должна позволять горизонтальное масштабирование компонентов.

**SCA06:** База данных должна поддерживать рост до 1000+ инструментов без потери производительности.

### Доступность

**AVA01:** Допустимый процент отказов системы (Uptime) должен составлять не менее 99% для прототипа и 99.5% для продакшена.

**AVA02:** Время недоступности при плановом обслуживании не должно превышать 4 часов в месяц.

**AVA03:** Время восстановления после сбоя (RTO) не должно превышать 2 часа.

**AVA04:** Точка восстановления данных (RPO) не должна превышать 24 часа.

### Безопасность и аудит

**SEC01:** Система должна соответствовать требованиям ФЗ-152 «О персональных данных».

**SEC02:** Обязательно шифрование данных при передаче (TLS 1.2 и выше).

**SEC03:** Обязательно хэширование паролей в базе данных.

**SEC04:** Обеспечить 100% защиту от SQL-инъекций.

**SEC05:** Обеспечить 100% защиту от XSS-атак.

**SEC06:** Обеспечить 100% защиту от CSRF-атак.

**SEC07:** Реализовать ограничение количества запросов (Rate Limiting) — не более 100 запросов в минуту с одного IP.

**SEC08:** Обязательное логирование всех действий администраторов.

**SEC09:** Минимизация сбора персональных данных до необходимого минимума.

### Надёжность 

**DUR01:** Обеспечить ежедневное резервное копирование базы данных.

**DUR02:** Хранить резервные копии не менее 30 дней.

**DUR03:** Обеспечить автоматическое восстановление после сбоев в течение 15 минут.

**DUR04:** Предусмотреть возможность ручного отката на предыдущую версию данных.

**DUR05:** Гарантировать сохранение пользовательских сессий при сбоях.

### Мониторинг

**MON01:** Реализовать мониторинг времени ответа API.

**MON02:** Реализовать мониторинг нагрузки на базу данных.

**MON03:** Реализовать мониторинг доступности сервиса.

**MON04:** Реализовать мониторинг ошибок и исключений.

**MON05:** Настроить автоматические алерты при падении доступности.

**MON06:** Обеспечить сбор анонимной статистики использования.

### Организационные и юридические вопросы 

**LRC01:** Разработать и опубликовать пользовательское соглашение.

**LRC02:** Разработать и опубликовать политику конфиденциальности.

**LRC03:** Обеспечить соответствие требованиям академической этики.

**LRC04:** Указывать источники информации об инструментах.

**LRC05:** Включить отказ от ответственности за точность рекомендаций.

### Локализация и интернационализация

**L10N01:** Обеспечить поддержку русского языка на момент запуска.

**L10N02:** Обеспечить поддержку английского языка на момент запуска (желательно).

**L10N03:** Предусмотреть возможность добавления новых языков без переписывания кода.

**L10N04:** Поддерживать форматы дат и чисел в зависимости от региона.

**L10N05:** Локализовать интерфейс, уведомления и справочные материалы.

### Целевые платформы и устройства пользователей

**PLT01:** Поддержка браузера Google Chrome (последние 2 версии).

**PLT02:** Поддержка браузера Mozilla Firefox (последние 2 версии).

**PLT03:** Поддержка браузера Safari (последние 2 версии).

**PLT04:** Поддержка браузера Microsoft Edge (последние 2 версии).

**PLT05:** Обеспечить адаптивный дизайн для мобильных устройств.

**PLT06:** Обеспечить адаптивный дизайн для планшетов.

**PLT07:** Обеспечить корректное отображение на десктопах.

### Ограничения

**LIM01:** В MVP поддерживается только русский язык интерфейса.

**LIM02:** База инструментов ограничена 20-25 наиболее популярными инструментами.

**LIM03:** Отсутствует интеграция с API вендоров в первой версии.

**LIM04:** Нет возможности автоматического обновления данных об инструментах.

**LIM05:** Ограниченная функциональность административной панели в MVP.

**LIM06:** Отсутствует мобильное приложение (только веб-версия).

### Принципы

**PRN01:** Прозрачность — все рекомендации должны иметь понятное обоснование.

**PRN02:** Объективность — рекомендации без рекламного влияния.

**PRN03:** Доступность — бесплатный базовый функционал для всех.

**PRN04:** Простота — интуитивный интерфейс для новичков.

**PRN05:** Актуальность — информация об инструментах должна быть свежей.

**PRN06:** Гибкость — архитектура для будущих расширений.

**PRN07:** Академичность — соответствие требованиям дипломной работы.

**PRN08:** Практичность — реальная польза для пользователей.
